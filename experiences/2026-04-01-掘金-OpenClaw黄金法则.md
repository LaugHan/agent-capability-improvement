## 核心

### 法则1: 模型分工（成本降低90%）

```yaml
# ❌ 不好：所有任务都用最贵的模型
model:
  default: gpt-4-turbo

# ✅ 好：按任务分配合适模型
model:
  default: deepseek-chat      # 普通对话
  code: deepseek-coder        # 写代码
  complex: gpt-4-turbo        # 复杂推理
```

**原理**：不是所有任务都需要最强模型，简单的用便宜的，复杂的用强的。

### 法则2: 启用缓存（重复问题0消耗）

```yaml
cache:
  enabled: true
  type: redis
  ttl: 3600  # 缓存1小时
```

**原理**：用户常问相同问题，缓存后直接返回，不消耗token。

### 法则3: 限制历史（Token减少50%）

```yaml
# ❌ 不好：保留太多历史
memory:
  max_turns: 100  # 太长，上下文爆炸

# ✅ 好：智能截断+摘要
memory:
  max_turns: 20
  summary_threshold: 10  # 10轮后开始摘要
```

**原理**：历史太长会稀释关键信息，还会花更多钱。

### 法则4: 环境变量（安全）

```yaml
# ❌ 不好：API Key写代码里
providers:
  deepseek:
    api_key: "sk-xxx"

# ✅ 好：环境变量
providers:
  deepseek:
    api_key: ${DEEPSEEK_API_KEY}
```

### 法则5-7: 生产必备

```yaml
# 告警：服务挂了知道
alerts:
  - name: service-down
    condition: health_check_failed
    action: send-telegram

# 备份：防止数据丢失
heartbeat:
  tasks:
    - name: backup
      schedule: "0 2 * * *"

# 成本：知道花了多少钱
budget:
  daily_limit: 100
  alert_threshold: 80  # 80%时通知
```

### 法则8: 先测试后发布

```bash
# ❌ 不好：直接发布
openclaw skill publish my-skill

# ✅ 好：测试通过再发布
openclaw skill test my-skill
openclaw skill publish my-skill
```

## 启发

这些法则的核心思想：
1. **成本控制**：让贵的资源用在刀刃上
2. **稳定性**：监控、告警、备份
3. **渐进式**：先测试再上线

不只在 OpenClaw，这些原则适用于任何 Agent 系统。

## 实际用法

1. 配置新的 Agent → 先看有没有按任务分模型
2. 上线新功能 → 先在测试环境跑
3. 长期运行 → 监控成本，设置告警
4. 调优性能 → 先从缓存和历史截断入手
